# Perchy 项目复盘

一天时间从零构建的桌面宠物应用，从 Windows-only 演进到 Windows + macOS（新 + 旧）三平台分发。本文记录**我们踩过的每一个坑**、它们的**根因**、**如何解决**，以及从中学到的可迁移经验。

## 一、项目是什么

一只坐在你当前激活窗口标题栏上、跟着窗口移动的小猫，每 3 分钟随机换一只姿势（9 张英短小猫图片）。Windows 上窗口最大化时猫会变成一串会摆动的小鞭炮 —— 点它就把窗口还原。

**核心特点**：透明置顶、点击穿透、不抢焦点、不占任务栏 / Dock、支持自定义 PNG、热重载。

## 三、坑的清单（按发生顺序，按类别分组）

### 3.1 环境和工具链

#### 坑 1：Python 3.7 装不上 PyQt6

- **症状**：`pip install PyQt6` 失败，`PermissionError: [WinError 32] 另一个程序正在使用此文件`
- **原因**：PyQt6 官方最低要求 Python 3.9。Python 3.7 没有预编译 wheel，pip 尝试从源码编译，编译中间文件被杀毒/文件索引器锁住。而且 Python 3.7 自 2023 年 6 月已 EOL。
- **解决**：换 Python 3.13
- **教训**：*立项前先核对每个依赖的支持矩阵。* Python 版本 = 最短板。

#### 坑 2：沙盒装不了 Pillow / 用不了 pip

- **症状**：`pip install Pillow` 报 `Cannot connect to proxy. 403 Forbidden`；沙盒是 `INTEGRATIONS_ONLY` 网络模式
- **原因**：Kiro Web 沙盒禁 pip 外部下载
- **解决**：改用 Go 标准库的 `image/png` 写抠图工具。Go 有全套 stdlib，不需要装东西
- **教训**：*沙盒环境要提前评估工具链可用性。* 一个工具装不上就要另想办法，Go / C / Bash 通常能救急。

#### 坑 3：一直在错误的目录里跑 git

- **症状**：用户多次报 `fatal: Not a git repository (or any of the parent directories): .git`
- **原因**：命令提示符停在 `C:\Users\Haven_Win\`（家目录），实际仓库在 `C:\Users\Haven_Win\perchy\`
- **解决**：文档里明确写 `cd C:\Users\Haven_Win\perchy` 作为每条命令的前置
- **教训**：*给终端命令时永远先给 `cd`。* 提示符里没有 `perchy` 结尾就是没进对。

### 3.2 图像处理

#### 坑 4：白背景下猫身上有"空洞"

- **症状**：Windows 文件资源管理器（白背景）上出现的猫，深色部分（黑色瞳孔、暗部阴影）**看得见底下的按钮**，像抠图没抠干净
- **原因**：抠图脚本用了 alpha 渐变 `[alphaLo=4, alphaHi=32]`，本意是柔化边缘抗锯齿。但**猫是深灰色**，猫身内部像素亮度只有 15-25，被误判成"边缘"给了半透明 alpha（30%-70%）。深色 + 半透明在黑背景下看不出，白背景下就现形了
- **解决**：CC 阈值从 8 提到 22（更严格判定"这是猫肉"），所有 in-blob 像素直接 `alpha=255`（无条件不透明）。边缘有一点点硬但看不出
- **教训**：*"边缘"和"内部"必须清楚分离。* Alpha 渐变只能用于**真的是边缘**的像素，不是"看着颜色深"就当边缘。

#### 坑 5：预乘 alpha 让深色像素消失

- **症状**：抠新图（透明背景 PNG）时，猫的**眼球高光、瞳孔、深阴影、鼻头**都不见了
- **原因**：Go 的 `image.RGBA` 用**预乘 alpha**存储。透明像素预乘后 `RGB=(0,0,0)`；一个不透明纯黑像素（如瞳孔）预乘后**也是** `(0,0,0)`。用 `channelDistance` 判前景时区分不出，深色像素被当作背景丢掉。
- **解决**：改用 `image.NRGBA`（非预乘），并用 **alpha 通道**（而不是 RGB 距离）判前景 —— 只要 alpha > 20 就是猫
- **教训**：*每次做图像处理都要先问自己：这个格式是不是预乘？* Go 的 image 包里 `RGBA` 预乘、`NRGBA` 不预乘；PyQt 里默认也预乘；PIL 默认不预乘。

#### 坑 6：分割工具的背景检测不对

- **症状**：新图检测出的背景色是 `(0,0,0)` 但明明是透明的
- **原因**：同坑 5。透明像素的**预乘 RGB** 就是 `(0,0,0)`，看着像黑色背景，其实是 alpha=0
- **解决**：`detectBackground()` 里增加返回值 `bgTransparent: bool`。先看四角的 **alpha**，如果 alpha < 30 就判定为透明背景，切换到 alpha 通道过滤；否则用 RGB 距离
- **教训**：*把"是啥背景"和"背景什么颜色"分开成两个判断。*

### 3.3 UI 定位

#### 坑 7：猫像浮在标题栏上方，脚离窗口很远

- **症状**：`OVERLAP=30` 却看着爪子离标题栏还有几十像素
- **原因**：`QLabel` 固定 180×180，图片按 `KeepAspectRatio` 缩放后**居中对齐**。一只 180×102 的胖猫，图片上下各留 39 像素透明空白，视觉上看猫就"浮"了
- **解决**：先把 label 从 `AlignCenter` 改成 `AlignBottom | AlignHCenter`（底对齐）。**后来彻底改成**：Widget 尺寸跟着**缩放后图片的实际尺寸**走，没有多余透明空间
- **教训**：*不要在图片周围留看不见的空白。* 位置逻辑用**实际像素**，不用"最大边框"。

#### 坑 8：窗口最大化时猫"沉进"应用内容里

- **症状**：Windows 最大化窗口后，猫不在标题栏，反而**沉到工具栏内部**，占了一大块
- **原因**：最大化时窗口顶边 y=0，猫的目标 y = `0 - 180 + overlap` = 负数，被 `y = max(y, 0)` 夹回 0。但 widget 是 180 高的大盒子，里面的猫图（假设 102 高）底对齐后，头顶到 widget 顶部还有 78 像素**透明填充** —— 这块透明填充跑进了可视范围，把猫向下顶了 78 像素
- **解决**：Widget 缩到跟图片一样大（见坑 7 的解决办法），透明填充没了，猫头就贴到标题栏了
- **教训**：*"看不见的空白"在极端情况下会现形。* 尤其是屏幕边界夹取（clamp）时。

#### 坑 9：不同姿势的猫下沉深度不一致

- **症状**：切换到胖猫（450×229 缩放到 180×92）时陷进标题栏 60%，切到瘦猫（367×393 缩放到 168×180）陷进 33%
- **原因**：`OVERLAP = 60`（像素）对不同高度的图产生**不同的比例**
- **解决**：改成分数语义。`OVERLAP = 0.5` 表示"猫身高一半陷进去"，永远都是被标题栏正好劈成两半
- **教训**：*用户想要的是"半在里半在外"这样的**语义**，不是"60 像素"这样的**数字**。* 参数应该表达**意图**，实现层再换算。

#### 坑 10：最大化时鞭炮位置反复调不对

- **症状**：`BUTTON_HORIZONTAL_ANCHOR = 0.5`（标题栏中央）→ 用户说"往右"。改成 `0.85` → 说"再右"。改成右缘偏移 150px → 说"还是偏左"
- **原因**：不同浏览器（Chrome / Edge / Safari）的 min/max/close 按钮宽度**不一样**，Chrome 大约 90-100px，经典 Win32 应用大约 138px。用**比例**算错，用**固定右缘偏移**又对不同应用不一致
- **解决**：最终定 `BUTTON_RIGHT_INSET_PX = 100`（离窗口右边缘 100 像素），Chrome 上刚好挤在最小化按钮左边
- **教训**：*"贴在系统 UI 边上"这种需求，没有一个万能数字。* 提供 config，让用户微调。

#### 坑 11：跳动动画每帧被重置

- **症状**：加了跳动动画，但看着不跳
- **原因**：`_on_tick` 每 30ms 会调用 `_enter_button_mode()`，里面每次都 `self._button.move(0, BOUNCE_RANGE_PX)` 把按钮位置**重置**到基线
- **解决**：把 button-mode 的初始化分成**首次进入时**（跑一次）和**每 tick**（只做位置追踪）。跳动的 `move()` 在自己的 QTimer 里 30ms 一次，位置追踪不再重置按钮内部的位置
- **教训**：*状态机的 `enter/exit/tick` 要分清楚。* 每 tick 里做的事情要幂等，别覆盖动画状态。

### 3.4 代码 Bug

#### 坑 12：`_in_button_mode` AttributeError

- **症状**：`python main.py` 启动就崩，`AttributeError: 'PetWindow' object has no attribute '_in_button_mode'`
- **原因**：`__init__` 里 `ImageManager` 初始化时会立刻回调 `_on_image_changed`，但那个方法要读 `self._in_button_mode`，而这个属性**声明在** `__init__` **更靠后的位置**
- **解决**：把所有可变状态属性**声明提前**到 `__init__` 的最开头，紧跟着 config 加载
- **教训**：*任何在 `__init__` 内会被同步回调触发的方法，其读取的属性必须先声明。* 简单原则：**先声明状态，再连信号槽**。

#### 坑 13：`setQuitAtLastWindowClosed` 拼错

- **症状**：程序启动直接抛异常
- **原因**：正确名字是 `setQuitOnLastWindowClosed`（**On** 不是 **At**）
- **解决**：改拼写
- **教训**：*Qt 方法名容易拼错的坑，靠 IDE 或类型检查器发现。* 这次因为在沙盒里没 IDE 提示。

### 3.5 Git / 分支管理

#### 坑 14：Merge 冲突"吃掉"了 tracker 的功能

- **症状**：`main` 上 `pet_window.py` `import WindowState from active_window_tracker`，但 `active_window_tracker.py` 里根本没有 `WindowState` 类。运行必崩
- **原因**：合并 `feature/maximize-status-hint` 到 `main` 时：
  - Base commit = `898479c`（首次加了 WindowState 的 commit）
  - HEAD (main) = `dba9097`（revert 了 `898479c`，等于把 WindowState 删掉了）
  - MERGE_HEAD (feature) = `6eff174`（继承了 `898479c`）
  - Git 的三方合并判定：**"main 显式改过，feature 没改"** → 采用 main 的版本
  - 结果 `active_window_tracker.py` 里没了 WindowState，但 `pet_window.py`（我用 `--theirs` 强制取了 feature 版）**还在 import** 它
- **解决**：借着做 macOS 支持的机会，把 tracker 重构成一个包（`active_window_tracker/{__init__.py,_base.py,_windows.py,_macos.py}`），从 feature 分支的正确版本重建
- **教训**：*少用 revert。* 尤其别 revert 一个后来还要被合并回来的 feature commit。要撤销就删远端分支或用 `git reset --hard` 单独处理。

#### 坑 15：在错误的分支上做修改

- **症状**：修 `_in_button_mode` 的 fix 推到远端 `feature/maximize-status-hint`，覆盖了正确的代码
- **原因**：`git checkout main` 做 revert 后忘了切回 feature 分支，`git branch --show-current` 显示 `main`。我在 main 上做了 str_replace，然后 push 到 feature（工具接受了任意 remote_branch_name）
- **解决**：`git reset --hard` 恢复本地 main，切到 feature 分支，重新在**正确的代码**上应用 fix，用 `force-with-lease` 覆盖远端错误版本
- **教训**：*每次改代码前先 `git branch --show-current`。* 尤其在多分支操作后。

### 3.6 打包与分发

#### 坑 16：`stop.bat` 完全不管用（三重原因）

用户反复反馈 stop.bat 关不掉猫，最终查明是**三层叠加**的问题：

**a. 只杀 `pythonw.exe`，没杀 `python.exe`**
- 用户前台调试时用 `python main.py`，跑的是 `python.exe`
- 原 stop.bat 只找 `pythonw.exe`（后台模式），前台的一个都杀不掉

**b. `wmic` 在 Windows 11 24H2 之后默认没装**
- 原 stop.bat 用 `wmic process where "commandline like '%main.py%'"` 过滤
- Win11 24H2 (2024 年) 起 wmic 成了 "Feature on Demand"，默认不装 → wmic 命令直接失败，for 循环拿到空列表 → **杀不到任何进程**

**c. 没杀 `perchy.exe`（打包后的进程名）**
- 用户下载了 GitHub Release 的 PyInstaller 打包版
- 打包版的进程名是 `perchy.exe`，不是 `python.exe` / `pythonw.exe`
- 原 stop.bat 三种都不认，什么都杀不掉

**最终 stop.bat**：
```bat
REM 1) 打包版 perchy.exe（PowerShell + Get-Process）
REM 2) 源码模式 python.exe 且命令行含 main.py
REM 3) 源码模式 pythonw.exe 且命令行含 main.py
```

- **教训**：
  - **一个打包工具会改变进程名**，杀进程要考虑所有分发形态
  - **wmic 已经不能依赖**，用 PowerShell 更可靠
  - **测试要覆盖所有启动方式**（前台 / 后台 / 打包版）

#### 坑 17：Windows SmartScreen 拦截

- **症状**：用户下载 exe 双击，`"Windows 保护了你的电脑"`
- **原因**：exe 没花钱做代码签名（签名证书 $99-$500 一年）
- **解决**：文档明确写：**点"更多信息" → "仍要运行"**
- **教训**：*免费开源应用不签名很正常，但要在使用说明里正面处理。*

#### 坑 18：macOS 应该是 dmg 不是 zip

- **症状**：用户反馈"苹果笔记本不是 dmg 格式的吗？"
- **原因**：偷懒用 zip（CI 简单，跨平台一致），但 dmg 才是 Mac 原生分发格式
- **解决**：用 `hdiutil` 打 dmg，包含 `.app` + `assets/` + Applications 快捷链接
- **教训**：*尊重目标平台的分发习惯。* 每个平台都有约定俗成，别硬用一套。

#### 坑 19：macOS 10.15 用户不能装 v1.2.x

- **症状**："此应用程序要求 macOS 11.0 或更高版本"
- **原因**：PyQt6 硬性要求 macOS 11+（用了 Big Sur 才引入的 API）。用户的 Mac 是 10.15 Catalina，且是老机型，升不上更高的 macOS
- **解决**：加**并行的 legacy CI job**：Python 3.9 + PyQt5 + `MACOSX_DEPLOYMENT_TARGET=10.15` + `Info.plist:LSMinimumSystemVersion=10.15`。同一份源码，用 `qt_compat.py` 猴子补丁让 PyQt5 的 flat enum 伪装成 PyQt6 的 scoped enum
- **教训**：
  - *用户设备可能很老*（2012 年的 MBP 卡在 10.15）
  - *抽象层让一份源码支持两种依赖*（`qt_compat.py`）
  - *legacy 分发别单独用 branch 维护*，同一 main + 不同 CI job 更好维护

#### 坑 20：macOS bundle 里资源路径

- **症状**：把 `Perchy.app` 拖到 `/Applications` 后猫不出现
- **原因**：`main.py` 的 `_app_dir()` 找 `assets/pets` 的相对路径。macOS 上 `sys.executable` 指向 `.app/Contents/MacOS/Perchy`（深层路径），要爬 3 层出 `.app` 目录，再从**并列位置**找 `assets/`
- **解决**：`_app_dir()` 里判断 `sys.platform == "darwin"` 特殊解析 `.app` 路径。dmg 里的 `assets/` 放在 `.app` 同级
- **教训**：*打包后的路径和源码路径不同*。macOS 的 `.app` bundle 更是嵌套三层深。

### 3.7 分发和用户体验

#### 坑 21：桌面上有 2 只猫

- **症状**：用户截图显示两只猫叠在一起
- **原因**：用户既双击了 `run.bat`（启动一个后台 pythonw），又后来手动跑了一次 `python main.py`（启动第二个前台进程），或者 `run.bat` 快捷方式在 `shell:startup` 里自动启了一份
- **解决**：文档提醒别混用启动方式；杀掉所有 `python*.exe` 后干净启动一次
- **教训**：*长期驻留的应用必须有单例锁*（比如 named mutex / 文件锁）。这次没做，欠账。

#### 坑 22：`git pull` 失败因为分支不存在

- **症状**：用户 `git pull` 报 `no such ref was fetched`
- **原因**：用户本地在 `feature/maximize-status-hint` 分支，但这个分支我在 sandbox 侧合并到 main 后已经删了远端
- **解决**：让用户 `git checkout main && git pull`，删本地的孤儿分支
- **教训**：*删远端分支前要考虑还在这个分支上的用户。* 或者干脆不删，让它废弃在那。

#### 坑 23：Kiro 沙盒的 `/tmp` 每次命令清空

- **症状**：用 `/tmp/split_cats` 编译 Go 工具，报 "no such directory"
- **原因**：Kiro Web 沙盒里 `/tmp` 是**每次 execute_bash 独立的**（不持久），只有 `/projects/sandbox/` 是持久化的
- **解决**：所有临时工具放 `/projects/sandbox/_tools/`
- **教训**：*先熟悉沙盒的持久化边界。*

## 四、关键设计决策

回顾整个项目，这些**主动的设计选择**证明是对的：

1. **按平台派发的 tracker 包**（`active_window_tracker/_windows.py` + `_macos.py`）—— 不是运行时 `if` 判断，而是**导入时**选择实现。加平台成本低。

2. **`qt_compat.py` 猴子补丁**让 PyQt5 装成 PyQt6 —— 一份代码支持两个 Qt。**关键洞察**：PyQt5 的 flat enum 可以通过 `Qt.WindowType = Qt` 假装成 PyQt6 的 scoped enum。

3. **OVERLAP 从像素改成分数** —— 参数表达**语义**（"半在里半在外"）不是数字（"60px"）。用户体验一致性 up.

4. **猫模式 vs 按钮模式的双状态机** —— 最大化窗口时切换到不同的 UI，共用一个 Widget，通过 Win32 style 位切换点击穿透

5. **CI 三 job 并发**（Windows / macOS 现代 / macOS legacy）—— 一次 tag push 出所有产物

6. **图片文件夹热重载** —— 每 10 秒扫一次 `assets/pets/`，用户不用重启程序换图

7. **抠图工具用 Go 而不是 Python** —— 沙盒里没 Pillow，Go 的 stdlib 就够用了。**一个平台无法用的时候看另一个平台**。

## 五、如果重新做一次

按重要性排序：

1. **一开始就设计跨平台架构**。第一版直接把 `ctypes.windll` 写进 `pet_window.py`，导致后面加 macOS 时到处 `if IS_WINDOWS:` 打补丁。应该**从第一天**就有抽象。

2. **用 TOML / JSON 存 config，不要 Python 文件**。`config.py` 打包进 exe 后**用户改不了**（改本地不生效）。TOML 就能被 exe 在运行时读。

3. **加托盘图标 + Quit 菜单**。用户不用去任务管理器杀进程，也就不会有反复 stop.bat 关不掉的痛苦。

4. **加单例锁**（named mutex / lock file）。防止双开产生"2 只猫"。

5. **`OVERLAP` 一开始就用分数**。避免中途改语义、写向后兼容分支的代码。

6. **打包版给 exe / .app 签名**。付费但换用户第一次运行体验，值。

7. **legacy 分发从一开始就规划**。第一版就有 `qt_compat.py`，不用之后大改。

8. **macOS `assets` 打进 `.app` 内部 + 用户级 override 目录**。不要要求用户"把 assets 和 .app 一起拖到同一处"，Mac 用户不习惯这种耦合。
